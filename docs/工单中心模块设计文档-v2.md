## 1. 模块定位与本次重构目标

工单中心是小审 3.0 中连接项目组与交付承接方的统一协作枢纽。

现阶段的主要问题是：

- 现有实现默认“服务方 = 部门”，只能表达“交付中心”这一类单一承接主体
- 实际业务中还存在“业务一部后台”“业务二部后台”等承接方，它们在组织架构上未必是独立部门
- 现有“我的任务”“派单调度”页面与新的多承接方模型不兼容，信息架构过于按角色拆散
- “系统管理”中的“人员服务标签配置”更适合下沉到具体交付组织内部，而不是全局配置

本次重构目标：

1. 将“服务方”从“部门”升级为“交付组织”
2. 在工单中心内支持创建、管理多个交付组织
3. 让工单总览页同时承载项目组视角与交付视角的完整流程演示
4. 删除旧的“我的任务”“派单调度”页面，改为在总览和详情中完成核心流转
5. 保持 Apple 风格的清晰、简洁、高品质视觉体验

## 2. 核心模型：部门与交付组织解耦

### 2.1 为什么不能继续用“部门”表示服务方

真实组织架构中的部门主要用于：

- 企业微信组织同步
- 人员归属管理
- 基础权限边界

而工单系统中的承接主体更接近“业务运营单元”：

- 交付中心
- 业务一部后台
- 业务二部后台
- 其他专项承接组

这些承接主体可能：

- 依附于某个部门存在
- 由同一部门内不同成员负责
- 跨部门协作
- 随业务调整而频繁变化

因此，工单中心中需要新增独立模型：`交付组织（delivery organization）`。

### 2.2 模型关系

```text
部门（真实组织）
  └── 交付组织（工单承接单元）
        ├── 成员
        ├── 角色权限
        ├── 服务模块标签
        └── 承接工单
```

### 2.3 设计原则

| 原则 | 说明 |
|------|------|
| 组织架构不变 | 企业微信同步、组织人员管理仍以“部门”为主 |
| 承接关系独立 | 工单中的“服务方”全部指向“交付组织”，不再直接指向部门 |
| 页面内聚 | 交付组织的成员、角色、标签配置全部放在工单中心内完成 |
| 配置最小化 | 原型阶段固定 4 种预置角色，不开放自由角色设计 |
| 兼容现有原型 | 尽量复用当前工单总览、详情抽屉、系统管理的现有组件和样式 |

## 3. 角色体系与可见范围

### 3.1 项目组侧角色

| 角色 Key | 角色名 | 职责 |
|----------|--------|------|
| `proj_member` | 项目组成员 | 发起工单、上传需求资料、查看状态 |
| `proj_lead` | 项目负责人 | 验收通过、驳回、查看所属项目全部工单 |

### 3.2 交付组织侧角色

| 角色 Key | 角色名 | 职责 |
|----------|--------|------|
| `delivery_owner` | 交付负责人 | 管理交付组织成员、角色、权限、服务标签；查看交付组织全部工单；查看排期；分派工单 |
| `module_lead` | 服务模块负责人 | 查看所属服务模块工单池；查看该模块成员排期；在该模块工单池中分派工单 |
| `delivery_staff` | 交付正式员工 | 查看自己所属服务模块池的工单；处理分派给自己的工单 |
| `delivery_intern` | 交付实习生 | 只查看分派给自己的工单；提交验收前的交付物上传 |

### 3.3 可见范围规则

| 角色 | 工单可见范围 |
|------|-------------|
| `proj_member` | 自己发起的工单 |
| `proj_lead` | 所属项目工作区的全部工单 |
| `delivery_owner` | 当前交付组织承接的全部工单 |
| `module_lead` | 当前交付组织内，且 `serviceModule = 我负责模块` 的工单 |
| `delivery_staff` | 当前交付组织内，且 `serviceModule ∈ 我的服务标签` 的工单；以及分派给自己的工单 |
| `delivery_intern` | 当前交付组织内，且 `assignee = me` 的工单 |
| `sys_admin` | 全量工单 + 全量组织 |

### 3.4 角色与组织的关系

- 一个用户可加入多个交付组织
- 同一用户在不同交付组织中可拥有不同角色
- 服务模块负责人以“组织内角色属性”存在，不改变其原始部门归属
- 工单中的权限判断优先基于“当前演示身份 + 当前交付组织 + 当前工单模块”

## 5. 交付组织模块设计

### 5.1 页面定位

“交付组织”是本次新增的核心管理页面，用于替代旧“派单调度”和旧“系统管理 > 人员服务标签配置”的部分功能。

### 5.2 页面布局

```text
┌────────────────────────────────────────────────────────────────────┐
│ 左侧：交付组织列表              │ 右侧：组织详情                    │
│                                │                                  │
│ [ + 新建交付组织 ]              │ [组织概览卡]                     │
│ 交付中心                       │ [成员管理] [角色权限] [服务标签]  │
│ 业务一部后台                   │                                  │
│ 业务二部后台                   │ 选中子页后展示具体内容            │
└────────────────────────────────────────────────────────────────────┘
```

### 5.3 新建交付组织

弹窗字段：

| 字段 | 类型 | 规则 |
|------|------|------|
| 组织名称 | 文本 | 必填 |
| 交付负责人 | 可搜索下拉 | 默认当前用户；仅可搜索当前登录人所属部门成员 |
| 交付组织所属部门 | 只读字段 | 自动映射当前登录人员所属部门 |

创建成功后：

- 自动生成 `deliveryOrgId`
- 自动将交付负责人加入组织成员列表
- 默认赋予其 `交付负责人` 角色
- 进入该交付组织详情页

### 5.4 成员管理

交互借鉴项目工作区中的“项目成员”，但字段面向交付组织重构。

成员列表字段：

- 姓名
- 角色
- 服务模块标签
- 服务模块组长
- 手机号
- 加入时间
- 有效期
- 操作（编辑 / 移出 / 续期）

### 5.5 角色权限

该页不再开放自由创建角色，仅展示 4 个预置角色的能力说明与权限矩阵：

- 交付负责人
- 服务模块负责人
- 交付正式员工
- 交付实习生

权限分组建议：

- 组织管理
- 工单查看
- 工单分派
- 排期查看
- 文件上传
- 文件版本更新

### 5.6 交付人员服务标签配置

基于当前工单中心中“系统管理 > 人员服务标签配置”改造，目标表格如下：

| 姓名 | 角色 | 服务模块组长 | 数据 | 试算 | 报告 | 函证 | 底稿 |
|------|------|--------------|------|------|------|------|------|

规则：

- 删除“部门”列
- 原“职级”列改为“角色”列，填充预置角色
- 新增“服务模块组长”列，为单选项：`数据 / 试算 / 报告 / 函证 / 底稿`
- 每位成员的服务标签仍可多选
- 一个交付组织内，同一服务模块建议仅配置一个组长

权限含义：

- 勾选某个服务模块，表示该成员可查看该模块工单池
- “服务模块组长”表示该成员可对该模块池内工单执行分派

规则：

这样做的原因：

- 顶部业务 Tab 已经承担主要导航作用

| 服务模块 | 枚举标签 |

| 总所编码 | 精确映射到选中主体 |
| 财报类型 | 单体 / 合并 |

| 提交人 | 发起人 |
| 提交时间 | 系统记录 |

| 处理人 | 未分派显示“—” |

- 批量催办
- 导出

## 8. 二类工单状态机

### 8.1 外显状态

二类工单在演示版中统一使用以下外显状态：

- `待接单`
- `已接单`
- `待验收`
- `验收通过`
- `已驳回`

### 8.2 状态流转

```text
项目组提交
  ↓
待接单
  ↓（交付负责人 / 模块负责人分配工单）
已接单
  ↓（交付人员上传交付物并点击“提交验收”）
待验收
  ├─（项目组点击“验收通过”）→ 验收通过
  └─（项目组点击“驳回”）→ 已驳回
                                 ↓
                           交付人员修正并再次“提交验收”
                                 ↓
                               待验收
```

### 8.3 状态对应操作

| 状态 | 项目组可见操作 | 交付组织可见操作 |
|------|----------------|------------------|
| 待接单 | 查看 | 派单 |
| 已接单 | 查看 | 作业 / 上传交付物 / 提交验收 |
| 待验收 | 验收通过 / 驳回 | 查看 |
| 已驳回 | 查看 | 作业 / 上传新版本 / 提交验收 |
| 验收通过 | 查看 | 查看 |

### 8.4 一类 / 三类工单补充

- 一类工单继续承担资源意向和总体预排作用，列表独立在 `一类工单` Tab 中展示
- 三类工单也进入五个模块池之一，但作业内容更偏驻场/工作区协同
- 原型演示阶段，一类工单可保留简化的“派单-处理-验收”流程，不再强制独立建“资源预排”页面

## 10. 演示模式设计

### 10.1 目标

你需要在“工单总览”页模拟项目组与交付的整体流程，因此保留“演示上下文”，但不再使用丑陋的“测试角色”条。

### 10.2 建议形式

页面右上方放两个轻量胶囊下拉：

- `演示身份`
- `交付组织`

#### 演示身份选项

- 项目组成员
- 项目负责人
- 交付负责人
- 服务模块负责人
- 交付正式员工
- 交付实习生

#### 交付组织选项

## 1. 模块定位与设计原则

### 1.1 模块定位

工单中心是小审 3.0 中连接**项目组（需求发起方）**与**交付中心（执行方）**的核心协作枢纽。

- 对项目组而言：是追踪所有委外任务进度、上传原料、验收成果的统一入口
- 对交付中心管理员而言：是完成资源预排、派单调度、人力看板的管理控制台
- 对交付中心一线人员而言：是领取任务、查看需求、上传成果、沟通进展的作业台

### 1.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **角色即视图** | 同一个页面，不同角色看到的工单范围、操作按钮、Tab 内容均不同，无需手动切换"视角" |
| **三层任务粒度** | 一类（资源意向）→ 二类（标准件交付）→ 三类（深度驻场），三层对应不同的调度逻辑 |
| **最小信息原则** | 工单列表行仅展示决策必要字段；详情由抽屉承载，避免列表过度信息密度 |
| **账套强挂载** | 二类工单执行时，处理人必须通过工单内的账套快捷链接查阅，杜绝使用工单外未记录的数据源 |
| **操作不可逆提示** | 接单、提交交付、验收通过等不可撤销操作，均需二次确认弹窗 |

## 2. 角色体系与视图隔离规则

### 2.1 角色定义

| 角色 Key | 角色名 | 所属组织 | 核心职责 |
|----------|--------|---------|---------|
| `proj_member` | 项目组成员 | 审计项目组 | 发起工单、上传原料 |
| `proj_lead` | 项目负责人 | 审计项目组 | 工单撤回/编辑、终审验收、驳回 |
| `dc_exec` | 交付中心执行人 | 交付中心 | 接单、作业、上传交付物（一类/二类工单） |

### 5.2 操作按钮区设计（权限细化）

> 每个操作按钮均依据**当前登录角色 + 工单当前状态**动态渲染，不满足条件的按钮不渲染（而非 disabled 置灰），保持界面干净。

| 按钮 | 可见角色 | 触发状态 | 操作后状态 | 二次确认 |
|------|---------|---------|----------|---------|
| **撤回** | `proj_lead` / `proj_member`（仅本人发起） | `待接单` | `已关闭` | ✅ "撤回后工单关闭，不可恢复" |
| **派单** | `dc_admin` | `待接单` / `资源调度中` | `业务处理中`（同时指定处理人） | 弹出派单面板（见 §6.2） |
| **接单** | `dc_exec`（服务标签匹配） | `待接单` | `资源调度中` | ✅ "接单后发起人不可撤回" |
| **转入复核** | `dc_exec`（已接单） | `业务处理中` | `内部复核中` | ✅ |
| **复核通过** | `dc_reviewer` | `内部复核中` | `待验收确认` | ✅ |
| **复核驳回** | `dc_reviewer` | `内部复核中` | `业务处理中` | 弹出填写驳回理由弹窗 |
| **提交交付** | `dc_exec`（处理人）/ `dc_admin` | `业务处理中` / `驳回修改` | `内部复核中`（如配置复核）/ `待验收确认`（如免复核） | ✅ "提交后项目组将收到验收通知" |
| **驳回** | `proj_lead` / `proj_member`（发起人） | `待验收确认` | `驳回修改` | 弹出填写驳回理由弹窗 |
| **验收通过** | `proj_lead` / `proj_member`（发起人） | `待验收确认` | `已关闭` | ✅ "验收后工单关闭，不可再修改" |

**补充说明：**
- 「派单」与「接单」互斥：`dc_admin` 主动派单后，该工单不再出现在自主接单池。
- 一类工单**无「接单」操作**，仅支持「派单」，因为资源预排由管理员统筹。
- 三类工单**无「接单」操作**，仅支持「派单」（管理员将执行人加入工作区）。

#### 6.4.2 工单模板配置（复核节点差异化配置）

- 服务模块默认信息要求
- **是否启用内部复核节点**（按模块独立配置，`true` = 提交交付后先进 `内部复核中`，`false` = 直接进 `待验收确认`）
- 复核人指定方式：`自动匹配`（按服务标签）/ `管理员指定`
- 建议工期（用于模板参考与排期提示）
- 必要资料清单（提交时展示为 Checklist，缺项软提示）

## 11. 交互规则与防呆机制

### 11.1 接单互斥锁

同一工单同一服务模块的「接单」操作必须是原子性的：第一个点击「接单」的用户成功，其余用户的界面同步更新（该工单从任务池消失），防止并发接单。

### 11.2 派单人力负载提示

派单面板中，每个可选执行人旁展示"当前负载 N 个工单"（实时计算），管理员选择负载过高（如 ≥3）的人员时，弹出提示："该执行人当前工单较多，确认派单？"，可继续操作但有警示。

### 11.3 上传文件保护机制

- 需求方上传区的文件：交付中心成员只读，项目组可删除/追加（工单关闭前）
- 服务方上传区的文件：项目组只读，交付中心执行人可追加（自动版本化），不可删除历史版本（保留审计链）

### 11.4 工单关闭前的最后验证

点击「验收通过」时，系统检查：
1. 服务方上传区是否有至少一个文件（交付物非空）
2. 若无文件，提示："服务方尚未上传交付物，确认通过？"（非硬阻断，管理员可强制通过并记录日志）

### 11.5 任务池抢单上限

若系统管理员配置了"每人最大同时持有工单数"（如 5），执行人超过上限时，「接单」按钮 disabled，悬停显示"您当前已持有 5 个工单，请完成后再接单"。

### 11.6 一类工单资源预排完成前的防断链

一类工单状态=`待接单` 超过 N 天（可配置，默认 2 天）未完成预排，系统向 `dc_admin` 推送提醒："您有 1 个一类工单待资源预排，请尽快确认。"

### 11.7 驳回次数标记

工单被驳回 ≥2 次时，工单列表行中的工单标题前显示"🔴 反复驳回"标记，提醒管理员关注。

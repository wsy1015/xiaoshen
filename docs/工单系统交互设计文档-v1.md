## 1. 角色定义与权限矩阵

### 1.1 角色定义

| 角色 | 所属 | 职责定位 |
|------|------|----------|
| **项目组成员** | 一线审计项目组 | 建立工作区、发起工单、提供底层原料、验收交付成果 |
| **项目负责人** | 一线审计项目组 | 工作区管理、工单撤回/编辑（接单前）、验收终审 |
| **交付中心执行人** | 交付中心 | 接单、执行标准化作业、上传交付物 |
| **交付中心复核人** | 交付中心 | 内部编制→复核闭环，复核痕迹对项目组可见 |
| **派单调度员** | 交付中心 | 工单资源调度、抢单/派单管理 |

### 1.2 权限矩阵

| 操作 | 项目组成员 | 项目负责人 | 交付中心执行人 | 交付中心复核人 |
|------|:---:|:---:|:---:|:---:|
| 新建工作区 | ✓ | ✓ | — | — |
| 导入总所项目编码绑定 | ✓ | ✓ | — | — |
| 上传账套 | ✓ | ✓ | — | — |
| 账套关联配置 | ✓ | ✓ | — | — |
| 发起工单（一/二/三类） | ✓ | ✓ | — | — |
| 撤回工单（接单前） | 仅本人 | ✓ | — | — |
| 编辑工单（接单前） | 仅本人 | ✓ | — | — |
| 接单 | — | — | ✓ | — |
| 上传交付物 | — | — | ✓ | ✓ |
| 查看/下载交付物 | ✓ | ✓ | ✓ | ✓ |
| 编辑/覆盖交付物 | — | — | ✓（仅自己上传） | ✓ |
| 内部复核 | — | — | — | ✓ |
| 验收通过 | — | ✓ | — | — |
| 驳回工单 | — | ✓ | — | — |

> **单向交付权**：交付物一旦上传至工单，项目组仅可查看/下载，不可编辑或覆盖，防止原始交付物被污染。

## 2. 工单类型定义

### 2.1 一类工单（大工单 · 资源意向锁定）

- **发起时机**：项目计划阶段，正式执行前
- **核心目的**：锁定服务方资源，明确服务模块组合与预估工作量
- **特征**：一个一类工单可关联多个服务模块；作为二类/三类工单的父级工单（`parentOrderId`）
- **典型标题格式**：`一类工单-$主体公司名称-$数字编号（001）`
- **主体公司来源**：工作区基础信息中的单一主体公司（`entity` 字段），非全量绑定清单

### 2.2 二类工单（标准模块工单 · 柜台式交付）

- **发起时机**：执行阶段，针对具体可标准化的服务模块
- **核心目的**：项目组提供原料（账套/资料），服务方产出标准化结果（数据清洗、流水核查、函证等）
- **特征**：单次提单绑定单一服务模块；支持多家主体公司并行；支持抢单+派单双轨并行
- **依赖前置**：所选主体公司必须全部完成账套关联方可提交（硬阻断）

### 2.3 三类工单（底稿交付工单 · 驻场协同）

- **发起时机**：执行阶段，需交付人员进入工作区深度协同
- **核心目的**：邀请交付人员进入项目工作区，按规则编制底稿
- **v1.0 范围**：仅实现提单入口，完整流转逻辑后续版本展开

## 3. 核心数据字段规范

### 3.1 基础身份字段

| 字段名 | 类型 | 说明 | 来源 |
|--------|------|------|------|
| `id` | String | 系统全局唯一流水号 | 系统自动生成，格式：`WO{YYYYMMDD}{4位序号}` |
| `title` | String | 工单标题 | 一类工单自动生成（可编辑）；二/三类自动生成 |
| `workspace` | String | 所属项目工作区名称 | 发起时自动带入当前工作区 |
| `company` | String | 关联主体公司名称 | **一类**：工作区 `entity` 字段；**二类**：用户在 Step 3 所选公司列表 |
| `firmCode` | String | 总所项目编码 | **一类**：工作区 `totalCode` 字段；**二类**：仅取所选公司对应的绑定行编码 |
| `reportType` | String | 财报类型（合并/单体） | **一类**：工作区顶层字段；**二类**：所选公司对应绑定行（去重后拼接） |
| `projectManager` | String | 项目负责人 | **一类**：工作区顶层字段；**二类**：所选公司对应绑定行（去重后拼接） |

> ⚠️ **字段映射精度规则（v1.2 更新）**：二类工单的 `firmCode`、`reportType`、`projectManager` 三个字段，**仅从用户在 Step 3 勾选的主体公司中映射**，不得包含工作区其余公司的数据。一公司对应一条总所编码，不允许一家公司对应多条。

### 3.2 流转控制字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `woType` | Enum | 工单类型：`一类工单 / 二类工单 / 三类工单` |
| `parentOrderId` | String | 所属一类工单编号（二/三类填写，一类为空） |
| `serviceModule` | String / List | 服务模块枚举，见第 10 节 |
| `moduleDetails` | Object | 一类工单专属：每个服务模块的备注、期望时间、附件（见 6.1 Step 3） |
| `dataCarrier` | Object | 数据载体：在线账套 URL、本地账套缓存 ID 或工作区文件夹软链 |
| `serviceProvider` | String | 服务方：`交付中心 / 项目组后台` |

### 3.3 权限与追踪字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `submitter` | String | 提交人 |
| `submitTime` | DateTime | 提交时间，系统自动记录 |
| `expectedTime` | Date | 期望完成时间；**可选最早日期 = 提交当日 +3 天** |
| `handler` | String | 服务方实际执行人（接单后写入） |
| `status` | Enum | 见第 4 节状态机 |
| `priority` | Enum | `特急 / 紧急 / 普通` |
| `rejectReason` | String | 驳回原因（驳回时填写） |
| `revisionCount` | Number | 驳回次数，用于 `-Rn` 后缀生成 |

## 4. 工单状态机

```
[发起提交]
    │
    ▼
 待接单
    │──────────────────────────────────────────┐
    │ 服务方接单                                 │ 发起人撤回（接单前可操作）
    ▼                                          ▼
 资源调度中                                   已关闭
    │
    │ 派单完成
    ▼
 业务处理中
    │──────────────────────────────┐
    │ 提交内部复核                   │ 直接交付（无需复核时）
    ▼                              │
 内部复核中                         │
    │ 复核通过                       │
    ▼◄──────────────────────────────┘
 待验收确认
    │──────────────────────┐
    │ 项目组验收通过          │ 项目组驳回
    ▼                      ▼
 已关闭                  驳回修改（编号追加 -Rn）
                            │
                            │ 执行人修改后重新提交
                            ▼
                         待验收确认（循环）
```

### 状态说明

| 状态值 | 触发条件 | 可执行操作 |
|--------|----------|------------|
| `待接单` | 工单提交后 | 发起人：撤回；服务方：接单 |
| `资源调度中` | 服务方接单后 | 派单调度员：分配执行人 |
| `业务处理中` | 派单完成后 | 执行人：作业、上传交付物、提交交付 |
| `内部复核中` | 执行人提交内部复核 | 复核人：通过/打回 |
| `待验收确认` | 内部复核通过或直接交付 | 项目组：验收通过/驳回 |
| `驳回修改` | 项目组驳回 | 执行人：修改后重新提交 |
| `已关闭` | 项目组验收通过 或 发起人撤回 | — |

## 5. 场景 0：工作区建项与账套关联

> 已实现，本节为规范性描述，便于后续功能对齐。

### 流程步骤

```
1. 正式成员在"客户管理"维护客资
   └─ 填写：公司全称、所属一级集团（不建树形结构）

2. 项目组新建工作区
   └─ 选择：业务类型、项目年度

3. 总所项目编码绑定（第三步"总所关联"）
   └─ 下载模板 → 填写主体公司与总所项目编码的一对一映射 → 上传 Excel
   └─ ⚠️ 阻断逻辑：未完成此步骤，发起一/二类工单时第一步禁用"下一步"

4. 账套入库
   └─ 在"账套管理"上传源文件（支持在线账套 URL 或本地文件）

5. 账套关联
   └─ 进入工作区 → 点击"账套关联"
   └─ 对每家已绑定主体公司，选择账套版本（V1/V2…）+ 起止账期（必填）
   └─ 支持同一公司关联多个账套版本，支持同一账套被多家公司关联
```

## 6. 场景 1：一类工单流转

### 6.1 发起流程（3步表单）

```
项目作业区 → 点击"发起工单" → 选择"发起一类工单"
```

**第一步：项目信息确认**

> ⚠️ **v1.2 更新**：主体公司信息只展示工作区基础信息中的**单一主体**（`entity` 字段），而非全量绑定清单。映射字段来源如下：

| 字段 | 来源 | 是否可编辑 |
|------|------|------------|
| 公司名称 | 工作区 `entity` 字段（单一主体） | 只读 |
| 总所项目编码 | 工作区 `totalCode` 字段 | 只读 |
| 财报类型 | 工作区 `reportType` 字段 | 只读 |
| 项目负责人 | 工作区 `manager` 字段 | 只读 |
| 工单标题 | 自动生成：`一类工单-{entity}-001` | 可二次编辑 |
| 工单编号 | 系统自动生成 | 只读 |

> **阻断逻辑**：若工作区 `totalCode` 为空（未完成"总所关联"绑定），展示错误提示并禁用"下一步"按钮。

**第二步：选择服务方**

- 单选：`交付中心` / `项目组后台`

**第三步：选择服务模块与补充信息**

- 多选服务模块（见枚举第 10 节）
- 每个已选模块展开独立填写区：

| 子字段 | 规则 |
|--------|------|
| 备注说明 | 可选文字 |
| 期望完成时间 | 可选，精度：日；**最早可选日期 = 提交当日 +3 天** |
| 说明附件 | ✅ 可选，点击上传或拖拽；支持 PDF / Excel / Word / 图片，单文件 ≤ 20MB |

> **提交确认**：状态置为`待接单`，工单同步至「工单中心 → 工单总览」和「本项目作业区 → 工单管理」。

### 6.2 工单管理模块（项目作业区内）

> **布局规则（v1.2 新增）**：进入"工单管理"Tab 后，左侧文件目录侧边栏自动隐藏，内容区占满全宽。

展示内容与"工单总览"页一致，过滤条件为当前工作区，操作列：

| 操作 | 可用条件 |
|------|----------|
| 查看 | 任意状态 |
| 撤回 | 状态为`待接单`且操作人为发起人或项目负责人 |
| 验收通过 | 状态为`待验收确认`且操作人为项目负责人 |
| 驳回 | 同上，需填写驳回原因 |

### 6.3 服务方接单流程

```
交付中心 → 进入"工单中心"
  └─ 自动过滤：仅展示"服务方=交付中心"的工单
  └─ 点击具体工单 → 工单详情抽屉
  └─ 点击"接单"按钮（仅"待接单"状态可见）
     ├─ 状态更新为：资源调度中
     ├─ 发起人"撤回"权限立即关闭
     └─ 派单调度员分配执行人
```

### 6.4 提交交付与验收完结

```
执行人 → 工单详情 → 点击"提交交付"（业务处理中/资源调度中状态可见）
  └─ 状态更新为：待验收确认
  └─ 项目组收到通知

项目组 → 工单管理 → 进入工单
  └─ 验收通过 → 状态置为"已关闭"
  └─ 驳回 → 填写驳回原因 → 状态置为"驳回修改"，编号追加 -Rn
```

## 7. 场景 2：二类工单流转

### 7.1 发起流程（4步表单）

```
项目作业区 → 点击"发起工单" → 选择"发起二类工单"
```

**第一步：选择服务方**

- 单选：`交付中心` / `项目组后台`

**第二步：选择服务模块**

- 单选，枚举：`数据 / 试算 / 报告 / 制函 / 底稿 / 复核 / 归档`
- 选择"制函"后展示细分类型：`银行电子函证 / 往来函证`（必选）

**第三步：选择主体公司与细分配置**

| 字段 | 规则 |
|------|------|
| 主体公司 | 多选，来自工作区绑定清单；选中后系统**仅取所选公司**对应的总所编码、财报类型、项目负责人 |
| 细分服务（按模块） | 示例（制函-往来函证）：是否需要沟通服务（开关型） |
| 期望完成时间 | 必填，精度：日；**最早可选日期 = 提交当日 +3 天** |

> ⚠️ **字段映射规则（v1.2 明确）**：`firmCode`/`reportType`/`projectManager` 仅从勾选的主体公司行映射，多家公司时各自对应，不合并、不引入未选公司的数据。

**第四步：上传相关材料**

| 区块 | 规则 |
|------|------|
| 账套带入（硬阻断） | ⛔ **v1.2 更新**：系统检测所选主体公司是否已关联账套。若有公司未关联，红色错误提示「检测到 xxx 主体未关联账套，请先关联账套后再提交工单」，账套勾选框禁用，"提交工单"按钮禁用 |
| 账套带入（正常） | 默认勾选「使用当前工作区已关联账套」；提示：提交后账套快照将同步至服务方，后续更新不会自动同步 |
| 必传材料 | 「务必上传关联主体公司的征信报告 & 开立户清单」，不可为空，阻断提交 |
| 选传材料 | 其他必要补充资料，可为空 |

**提交**：状态置为`待接单`，同步到工单总览 + 本项目作业区工单管理。

### 7.2 服务方接单与闭环作业

```
接单 → 状态：资源调度中
  └─ 派单/抢单 → 状态：业务处理中
  └─ 执行人通过快捷链接查阅账套（在线跳转 / 本地唤起）
  └─ 完成标准结果 → 上传至工单"交付物区"
  └─ 提交交付 → 触发内部复核（若需）或直接进入"待验收确认"
```

### 7.3 验收完结

同一类工单，参见 6.4 节。

## 11. 前端交互规则与防呆机制

### 11.1 发起工单入口阻断

```
一/二类工单：工作区 totalCode 为空（未完成总所绑定）
  └─ 类型选择器中，一类/二类卡片点击时弹出提示
  └─ 提示：「请先完成工作区的总所项目编码绑定，再发起一/二类工单」
  └─ 三类工单不受此限制
```

### 11.2 账套关联硬阻断（v1.2 更新）

```
二类工单 Step 4 进入时
  └─ 系统检测：所选主体公司在工作区 assocMap 中是否均有账套关联
  └─ 若有未关联账套的公司：
     ├─ 红色错误提示框：「检测到 xxx、xxx 主体未关联账套，请先关联账套后再提交工单」
     ├─ 账套勾选框禁用（灰色，不可操作）
     └─ "提交工单"按钮禁用
  └─ 验证层：validateWoStep 兜底校验，防止绕过 UI 直接触发提交
```

> ⚠️ **与 v1.1 的差异**：原规范为软提示（弹窗 + 继续提交选项），v1.2 改为**硬阻断**，必须完成账套关联后才能进行提交。

### 决策 1：一类工单关联公司范围（v1.2 修订）

**原结论（v1.1）**：一类工单自动关联发起时所在项目工作区的全部公司名称，展示完整绑定清单表格。

**修订结论（v1.2）**：一类工单的主体公司信息只展示工作区基础信息中的**单一主体**（`entity` 字段），对应的总所项目编码取工作区 `totalCode`，财报类型和负责人取工作区顶层字段。

**修订原因**：一类工单是资源意向锁定，对应的是项目组的一个审计作业区（`entity`），而非绑定清单中全部子实体。多主体并行交付在二类工单中体现。

**规范影响（v1.2 更新）**：

| 字段 | 来源 |
|------|------|
| 公司名称 | 工作区 `entity`（单一主体，只读） |
| 总所项目编码 | 工作区 `totalCode`（只读） |
| 财报类型 | 工作区 `reportType`（只读） |
| 项目负责人 | 工作区 `manager`（只读） |
| 工单标题 | 自动生成，可二次编辑 |
| 工单编号 | 系统自动生成，只读 |

---

### 决策 2：二类工单抢单任务池分配规则

**结论**：任务池按**服务模块**拆分为独立子池，成员仅可见其持有对应服务标签的子池中的工单。  
服务方负责人在"工单中心 → 系统管理"中为下属成员分配**服务标签**（枚举与工单"服务模块"字段保持一致）。

**规范影响（新增）**：

#### 服务标签管理（系统管理 → 成员服务标签）

| 角色 | 可执行操作 |
|------|------------|
| 服务方负责人 | 为下属成员分配/撤销服务标签（多选） |
| 普通成员 | 查看自己的服务标签，不可自行修改 |

#### 任务池可见性规则

```
交付中心成员登录"工单中心"
  └─ 系统读取该成员的服务标签列表
  └─ 仅展示"服务模块 ∈ 该成员服务标签"且"状态=待接单"的工单
  └─ 若成员持有多个服务标签，可通过"服务模块"筛选项切换查看各子池
```

#### 抢单与派单并行规则

```
派单：服务方负责人/调度员在任务详情页指定执行人（忽略标签限制）
抢单：执行人在任务池列表中主动点击"抢单"（需有对应服务标签）
冲突处理：同一工单被抢单成功后，派单按钮置灰，已派单后不可再被抢单
```

### 决策 3：列表是否展示独立时效状态

**结论**：不再维护独立的时效状态字段，统一仅展示 `期望完成时间`，如需提醒则通过消息或通知能力承接。

---

### 决策 4：复核痕迹可见粒度

**结论**：项目组可见**全部复核批注**，包括批注内容、批注人、批注时间及所属版本。  
内部执行链路（编制→复核）的中间版本及批注痕迹均对项目组透明，增强交付过程可信度。

**规范影响（更新）**：工单详情页"复核记录"区块展示内容：

```
每条复核记录包含：
  ├─ 版本号（如 V1 复核、V2 复核）
  ├─ 复核人姓名 + 复核时间
  ├─ 复核结论（通过 / 打回）
  └─ 全部批注列表（批注内容 + 批注时间 + 批注人）
```

---

### 决策 5：服务方复用统一状态机

**结论**："项目组后台"与"交付中心"作为服务方时，**复用同一状态机**，不做分支处理。  
后续如需针对"项目组后台"特殊处理（如跳过内部复核步骤），以配置开关方式迭代，不修改状态机结构。

---

### 决策 6：驳回后工单编号规则

**结论**：驳回后重新提交**复用原工单编号**，以 `-Rn`（`n` 为驳回次数）作为版本后缀，不新建工单。

**示例**：

```
原工单：WO202603060001
第1次驳回修改后：WO202603060001-R1
第2次驳回修改后：WO202603060001-R2
```

> 工单总览列表中展示最新版本编号；工单详情页"操作日志"中完整记录所有版本的驳回原因与修改记录。
